# MCP Connection Audit, 2026-06-08

A focused audit of Scalar's MCP server: can external agents (Claude custom
connectors and the Messages API `mcp_servers` beta, Managed Agents via vaults,
and key-based agents like OpenClaw and Hermes) connect, authenticate, and
operate the CRM with the right tools? Verdict: **connectivity is sound; the tool
set had real gaps, three of which are now closed.**

---

## 1. Connectivity and protocol (PASS)

| Requirement | Status | Where |
|---|---|---|
| Streamable HTTP transport at one URL | ✅ `https://www.tryscalar.xyz/api/mcp/mcp` | `src/app/api/mcp/[transport]/route.ts` (mcp-handler 1.1.0, MCP SDK 1.29.0) |
| Bearer token auth (key-based agents) | ✅ per-user API keys `scl_...`, SHA-256 | `withMcpAuth` + `api-auth.ts` |
| OAuth 2.1 + PKCE (S256) | ✅ authorize/token, Clerk-gated identity | `api/oauth/*`, `lib/oauth.ts` |
| Dynamic Client Registration (RFC 7591) | ✅ public clients, `token_endpoint_auth: none` | `api/oauth/register` |
| Authorization Server Metadata (RFC 8414) | ✅ `/.well-known/oauth-authorization-server` | `api/oauth/metadata/authorization-server` |
| Protected Resource Metadata (RFC 9728) | ✅ 401 points here, advertises the AS | `api/oauth/metadata/protected-resource` |
| `refresh_token` grant (vault auto-refresh) | ✅ supported on the token endpoint | `api/oauth/token` |
| Public routes (no Clerk wall on MCP/OAuth) | ✅ `/api/mcp`, `/api/oauth`, `/.well-known` | `src/proxy.ts` |

This matches what Claude's connector stack expects: a Streamable HTTP server
that either takes a static bearer token or runs the OAuth dance, with the
discovery metadata so the client can find the authorization server. Managed
Agents vaults need exactly the `refresh_token` grant with a public-client
(`none`) token-endpoint auth method, which the token endpoint provides.

### How each agent connects

- **Claude custom connector / Messages API `mcp_servers` / Claude Cowork** -
  add the URL `https://www.tryscalar.xyz/api/mcp/mcp`. The 401 advertises the
  protected-resource metadata, the client registers (DCR) and runs PKCE, the
  user signs in to Scalar to authorize, and the access token flows on
  `Authorization: Bearer`. No key to paste.
- **OpenClaw / Hermes / key-based or headless agents** - create a key in
  Settings -> API keys and send `Authorization: Bearer scl_...` to the same URL.
  This is what the bundled Claude Code plugin uses (`plugins/scalar/.claude-plugin/plugin.json`).
- **Managed Agents (CMA)** - declare the server on the agent
  (`mcp_servers: [{type:"url", name:"scalar", url:".../api/mcp/mcp"}]`) and put
  an `mcp_oauth` credential in a vault (access + refresh token, `token_endpoint`
  = `/api/oauth/token`, `token_endpoint_auth: {type:"none"}`). Anthropic
  auto-refreshes.

### Smaller protocol notes (not blockers)

- **Resource Indicators (RFC 8707):** `/authorize` and `/token` ignore the
  `resource` parameter. The protected-resource metadata sets the resource URL,
  and real connectors interoperate, but strict audience-binding is not enforced.
  Low risk; revisit if a client rejects the token audience.
- **OAuth signing secret** falls back to `CLERK_SECRET_KEY` when
  `MCP_OAUTH_SECRET` is unset (intentional per #17). Setting a dedicated secret
  is cleaner.

## 2. Tool coverage

The server exposes a shared ops layer (`crm-operations`, `field-operations`) so
REST, the in-app agent, and MCP behave identically. Before this audit: 23 tools.

### What an agent could already do
Entities (list/get/create/update/enrich), Contacts (list/get/create/update),
email context, `search_crm`, `search_web` (Tavily), Segments (list/get/create/
build_smart), Pipelines (list/get/create/add/update_entry/metrics).

### Gaps found, and what changed

| Gap | Severity | Action |
|---|---|---|
| **No `find_companies`** - the flagship "discover CRM-ready companies from a prompt" (Exa) was not exposed; only raw `search_web`. An agent could not do the core prospecting loop. | High | **Added.** `find_companies` wraps `exaFindCompanies`, dedupes by domain then name against the CRM and within the batch, and adds the new entities. |
| **No delete tools** - `deleteEntity` / `deleteContact` ops existed but weren't exposed, so an agent could create junk but never clean it up. | High | **Added** `delete_entity` and `delete_contact` (marked destructive). |
| **No tool annotations** - clients couldn't tell read-only from destructive tools, so they can't safely parallelize reads or gate deletes. | Medium | New tools carry `readOnlyHint` / `destructiveHint` / `idempotentHint` / `openWorldHint`. Backfilling the existing 23 is recommended (below). |

### Still open (recommended next, ranked)

1. **`enrich_contact`** (find verified email / phone / LinkedIn per field). This
   is core and currently lives only in the REST route
   (`api/contacts/[id]/enrich`), with the provider waterfall and the hard
   accuracy rule (name AND company/domain match). Exposing it via MCP means
   extracting that logic into a shared `crm-operations` op first - worth doing
   carefully, not hastily, because a wrong contact must never be saved.
2. **`spawn_contacts`** - research a company's decision-makers and add the ones
   not already present (`exaResearchContacts` exists; route logic in
   `api/entities/[id]/spawn-contacts`).
3. **`match_contact_to_entity`** - link an unassigned contact to its company.
4. **Intent monitors over MCP** - create/list/delete scheduled intent scans, so
   an agent can set up its own recurring discovery.
5. **Backfill annotations** on the existing 23 tools: `readOnlyHint: true` on
   every `list_*` / `get_*` / `search_*` / `*_metrics`, write hints on the rest.

## 3. Verification done

`tsc`, `eslint`, and `next build` pass with the three new tools and the new
`findCompanies` op. The OAuth handshake and a live connector round-trip still
need to be exercised against a real client (owed to reality) - the static
surface is correct, but only a real Claude connector or `ant beta:vaults`
credential proves the end-to-end flow.
